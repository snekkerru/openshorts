# «Мои генерации» (My Generations) — Design

**Date:** 2026-07-24
**Status:** Approved

## Problem

AI Shorts generations are ephemeral. `saas_jobs` is an in-memory dict wiped on
backend restart; the frontend persists only the single *current* job in
localStorage (`SESSION_KEY`); finished videos on disk are swept after
`JOB_RETENTION_SECONDS = 3600` (1h). So a user who closes the tab mid-generation,
or wants to revisit a past video, has no way to find it.

## Goal

A new left-nav section **«Мои генерации»** that lists the user's video
generations — completed history plus in-progress ones (including jobs started
then closed) — backed by durable S3 storage so it survives restarts and the
1-hour disk sweep.

## Decisions (confirmed with user)

- **Storage:** Backend + S3 (private bucket `AWS_S3_BUCKET`), mirroring the
  existing `upload_video_to_gallery` / `list_video_gallery` pattern but under a
  private, per-user `my-generations/{owner}/` prefix served via **presigned
  URLs** (not public `*.amazonaws.com` URLs — so it works on non-AWS S3).
- **S3 provider:** DigitalOcean Spaces (S3-compatible). Requires a custom
  `endpoint_url`; see "S3 client" below.
- **Multi-tenant:** each user's generations live under their own folder
  `my-generations/{owner}/…`. `owner` = the cloud user id from `_owner_id()`, or
  the literal `"local"` in self-host (billing off, no auth). List/delete are
  scoped to the requesting owner, so future multi-user deployments isolate
  histories automatically.
- **Always recorded:** every generation is saved to history automatically (NOT
  gated behind the opt-in `share_to_gallery`).
- **Open completed:** a dedicated lightweight viewer *inside* the section
  (modal), not the wizard's Result step. Player + metadata + download + copy.
  No publish-to-social from here.
- **Old on-disk videos:** ignored. History starts fresh from new generations.
- **Retention untouched:** the S3 copy is authoritative and durable; the local
  disk sweep stays as-is. Retry after the 1h window regenerates from the stored
  script rather than disk cache.

## Architecture

Three layers, each with one responsibility:

1. **`s3_uploader.py`** — persistence primitives for the private history prefix.
2. **`app.py`** — record lifecycle hooks in `run_generation` + list/delete
   endpoints that merge durable S3 records with live in-memory job status.
3. **`dashboard/src/components/MyGenerationsTab.jsx`** — the section UI + viewer
   modal; a nav entry in `App.jsx`.

### S3 client (DigitalOcean Spaces)

`get_s3_client()` currently builds an AWS-only client (no `endpoint_url`). Add
optional endpoint support, backward-compatible (unset → identical AWS behavior):

```python
endpoint = os.environ.get('AWS_S3_ENDPOINT_URL')  # e.g. https://fra1.digitaloceanspaces.com
boto3.client('s3', ..., endpoint_url=endpoint or None,
             config=Config(signature_version='s3v4'))
```

DO Spaces region is the datacenter slug (e.g. `fra1`) in `AWS_REGION`; the bucket
is the Space name in `AWS_S3_BUCKET`. Presigned GET URLs generated by boto3 point
at the configured endpoint automatically, so no URL string-building is needed for
history. The existing public-gallery functions build `*.s3.{region}.amazonaws.com`
URLs by hand — those are **out of scope** and untouched (history never uses them).

### S3 layout (private bucket, `AWS_S3_BUCKET`, per user)

```
my-generations/{owner}/{job_id}/metadata.json   # always
my-generations/{owner}/{job_id}/video.mp4        # on completion
my-generations/{owner}/{job_id}/actor.png        # on completion (thumbnail)
```

`owner` = `_owner_id(request)` (cloud user id) or `"local"` for self-host. A
small helper `_history_owner(request)` in app.py returns `await _owner_id(request)
or "local"`.

`metadata.json` schema:

```json
{
  "job_id": "uuid",
  "status": "processing | completed | failed",
  "title": "script title",
  "caption": "…",
  "hashtags": ["#…"],
  "full_narration": "…",
  "duration": 22,
  "video_mode": "lowcost | premium | maximum",
  "language": "ru | en",
  "cost_estimate": { "total": 2.55, "...": 0 },
  "product_name": "",
  "error": "message when status=failed, else \"\"",
  "created_at": "ISO-8601Z",
  "updated_at": "ISO-8601Z"
}
```

### Backend: `s3_uploader.py`

All three take `owner` and scope to `my-generations/{owner}/`:

- `save_generation_record(owner, job_id, metadata, status, video_path=None, actor_image_path=None)`
  → writes/overwrites `my-generations/{owner}/{job_id}/metadata.json` on the
  private bucket; uploads `video.mp4` / `actor.png` when the paths are given and
  exist. Sets `status`, `updated_at` (and `created_at` on first write, preserved
  on later writes by reading the existing record). Returns bool. Degrades to a
  no-op returning False when `get_s3_client()` is None (no S3 configured).
- `list_my_generations(owner, limit=100)` → scan `my-generations/{owner}/`, read
  each `metadata.json`, attach fresh presigned URLs `video_url` / `actor_url` (2h
  expiry) for the objects that exist, return newest-first (by `created_at`).
- `delete_my_generation(owner, job_id)` → delete every object under
  `my-generations/{owner}/{job_id}/`. Returns bool.

Reuse `get_s3_client()` (now endpoint-aware) and `generate_presigned_url()`
already in the module.

### Backend: `app.py`

`owner` is resolved once per request via `_history_owner(request)`
(`await _owner_id(request) or "local"`) and passed to the s3 helpers. The
generation task captures `owner` at submit time so its start/completion/failure
writes all land under the same folder.

Lifecycle hooks in `run_generation` (the async task in `saasshorts_generate`):

- **At job start** (right after `saas_jobs[job_id]` is created, for both fresh
  and retry paths): `save_generation_record(owner, job_id, meta_from_script, "processing")`
  where `meta_from_script` is built from `req.script` + `req.video_mode`.
  Wrapped in try/except so an S3 hiccup never blocks generation.
- **On completion:** `save_generation_record(owner, job_id, meta, "completed",
  video_path=result["video_path"], actor_image_path=result.get("actor_image"))`.
  This runs regardless of `share_to_gallery` (that opt-in gallery upload stays
  separate and unchanged).
- **On failure:** `save_generation_record(owner, job_id, meta, "failed")` with
  the error string in `metadata["error"]`.

Endpoints:

- `GET /api/saasshorts/my-generations` → `list_my_generations(owner)`, then merge
  with the caller's live `saas_jobs`. "Caller's own" is decided by a
  `_job_belongs(job, request)` predicate: `True` for all jobs when
  `BILLING_ENABLED` is off (single self-host user), else `job["user_id"] == await _owner_id(request)`.
  (Note the folder name `"local"` is only an S3 path token; the in-memory
  self-host `user_id` stays `None` — the predicate, not string equality, links
  them.) Merge rules:
  - If a listed `job_id` is live in `saas_jobs` (and belongs to caller), the live
    `status` wins (covers a job that started `processing` in S3 and is now
    `completed`/`failed` in memory before the S3 record is overwritten — rare race).
  - If an S3 record is `processing` but the `job_id` is NOT in `saas_jobs`
    (server restarted, task died): report `status: "interrupted"`.
  - Newest-first. Returns `{ "generations": [...] }`.
- `DELETE /api/saasshorts/my-generations/{job_id}` → `delete_my_generation(owner,
  job_id)`; also drop it from `saas_jobs` if present and owned. Returns
  `{ "deleted": true }`.

Auth: same policy as the sibling saas endpoints (self-host: open; cloud:
`require_managed_entitlement`). Scoping by `owner` means a cloud user only ever
lists/deletes their own folder; in self-host there is a single implicit `"local"`
owner.

### Frontend: `MyGenerationsTab.jsx`

- On mount and every 5s **while any item is `processing`/`interrupted`**, fetch
  `GET /api/saasshorts/my-generations`.
- **Card grid**, newest-first. Each card:
  - actor thumbnail (or placeholder), title, `created_at` (localized), cost.
  - status badge: `processing` (spinner, "в процессе"), `completed` ("готово"),
    `failed` ("ошибка"), `interrupted` ("прервана").
  - actions by status:
    - completed → **Открыть** (viewer modal), **Скачать**, **Удалить**
    - processing → **Открыть** (progress: poll `GET /api/saasshorts/status/{job_id}`
      and show the existing generation-log view), **Удалить**
    - failed / interrupted → **Повторить** (re-submit `POST /api/saasshorts/generate`
      with the stored script + `retry_job_id`), **Удалить**
- **Viewer modal** (completed): `<video>` player from presigned `video_url`,
  title, caption + copy button, hashtags + copy, cost breakdown, duration,
  Download button. No social publish.
- Retry uses the existing generate endpoint; the stored `metadata.json` carries
  the script needed to regenerate even if the disk cache was swept.

### Frontend: `App.jsx`

- Add nav item `{ id: 'my-generations', ord: '02', icon: <History/>, label: 'My Generations' }`
  between `saasshorts` and `settings`; renumber `settings` ord to `03`.
- Render `<MyGenerationsTab .../>` when `activeTab === 'my-generations'`, passing
  the fal/elevenlabs/openrouter keys + image-model props it needs for Retry
  (same props already threaded to `SaaShortsTab`).
- i18n: add the section's visible strings to `lib/i18n.js` (RU) and wrap via
  `t()`; nav label `'My Generations'` → `'Мои генерации'`.

## Error Handling

- S3 unavailable (no AWS keys / client None): `save_generation_record` and
  `list_my_generations` degrade to no-op / empty; generation itself is
  unaffected. The section shows an empty state ("нет генераций" / a hint that S3
  is not configured). This matches the user's choice of an S3-backed history.
- Presigned URL generation failure for one object: that field is omitted; the
  card still renders (e.g. no thumbnail).
- Delete of a non-existent prefix: idempotent success.

## Testing

- `tests/test_my_generations.py` (pytest, mock the S3 client):
  - `save_generation_record` writes to `my-generations/{owner}/{job_id}/…` with
    status/timestamps; preserves `created_at` on re-write; no-op when client None.
  - `list_my_generations(owner)` returns newest-first, attaches presigned URLs,
    and only reads the owner's prefix (a second owner's records are not returned).
  - merge logic: live status overrides S3; S3 `processing` absent from memory →
    `interrupted`. (Unit-test the pure merge helper extracted from the endpoint.)
- Frontend: manual e2e (Playwright) — section lists a completed generation,
  opens the viewer, downloads; a `processing` item shows the spinner; delete
  removes the card. Lint must pass.

## Configuration (env)

- `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` — DO Spaces key + secret.
- `AWS_REGION` — DO datacenter slug (e.g. `fra1`).
- `AWS_S3_BUCKET` — the Space name (private history lives here).
- `AWS_S3_ENDPOINT_URL` — **new**, e.g. `https://fra1.digitaloceanspaces.com`.
  When unset, the client stays AWS-compatible (no behavior change).

## Out of Scope

- Migrating the 3 pre-existing on-disk generations.
- Publishing to social from the viewer (already available in the wizard Result
  step).
- Pagination beyond a simple `limit` (100 newest is plenty for a personal tool).
- Changing `JOB_RETENTION_SECONDS` or the public gallery.
